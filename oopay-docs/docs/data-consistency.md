# OOPay 数据一致性保障清单

## 1. 一致性风险分析

### 1.1 核心业务场景风险

| 场景 | 风险描述 | 影响级别 |
|-----|---------|---------|
| **重复下单** | 商户重复提交同一订单 | 资金损失 |
| **重复回调** | 上游多次回调同一订单 | 重复结算 |
| **额度扣减异常** | 订单成功但额度未扣 | 资金损失 |
| **额度释放异常** | 超时订单额度未释放 | 额度浪费 |
| **状态不一致** | 平台/上游/商户状态不同 | 客诉/资金损失 |
| **缓存DB不一致** | 缓存额度与DB不符 | 超卖/拒绝服务 |
| **重复结算** | 同一订单多次结算给商户 | 资金损失 |

---

## 2. 数据一致性保障措施

### 2.1 接口层

| 措施 | 实现方式 | 应用场景 |
|-----|---------|---------|
| **幂等Token** | 商户请求携带唯一token，平台记录处理结果 | 统一下单接口 |
| **请求去重** | 基于商户ID+商户订单号做去重窗口（5分钟） | 防止重复下单 |
| **限流防重** | 同一商户1秒内同一订单号只允许1次 | 接口防刷 |

**实现细节：**
```java
// 幂等Key生成
String idempotentKey = "idempotent:" + mchId + ":" + mchOrderNo;

// Redis SETNX 5分钟过期
Boolean success = redis.setnx(idempotentKey, "processing", 300);
if (!success) {
    // 查询已处理结果返回
    return getPreviousResult(idempotentKey);
}
```

---

### 2.2 业务层

| 措施 | 实现方式 | 应用场景 |
|-----|---------|---------|
| **状态机校验** | 订单状态只能按既定路径流转 | 订单状态管理 |
| **分布式事务** | Saga模式：本地事务+补偿 | 支付流程 |
| **预扣机制** | 先冻结额度，成功后再扣减 | 额度管理 |
| **异步对账** | 订单完成后异步核对三方状态 | 状态一致性 |

**订单状态机：**
```
生成(0) → 支付中(1) → 成功(2) → 完成(3)
  ↓           ↓            ↓
关闭(4)    关闭(4)      关闭(4)

允许流转：
0→1, 0→4, 1→2, 1→4, 2→3, 2→4
禁止：任意状态回退
```

**Saga事务流程：**
```
下单流程：
1. 创建订单（本地事务）
2. 冻结额度（本地事务，失败则关单补偿）
3. 调用上游（外部调用，失败则释放额度补偿）
4. 更新订单（本地事务）

补偿机制：
- 冻结失败 → 关单
- 上游调用失败 → 释放额度 + 关单
- 回调处理失败 → 记录差异，人工介入
```

### 2.3 数据层

| 措施 | 实现方式 | 应用场景 |
|-----|---------|---------|
| **唯一索引** | 商户ID+商户订单号联合唯一 | 防重复订单 |
| **平台订单号唯一** | pay_order_id全局唯一 | 订单标识 |
| **乐观锁** | version字段控制并发更新 | 额度更新 |
| **悲观锁** | SELECT FOR UPDATE | 结算处理 |

**额度更新乐观锁：**
```sql
UPDATE provider 
SET available_quota = available_quota - #{amount},
    frozen_quota = frozen_quota + #{amount},
    version = version + 1
WHERE id = #{id} 
  AND available_quota >= #{amount}
  AND version = #{version}
```

**关键唯一索引：**
```sql
-- 防重复订单
UNIQUE KEY uk_mch_order (merchant_id, mch_order_no)

-- 平台订单号唯一
UNIQUE KEY uk_pay_order_id (pay_order_id)

-- 商户应用编码唯一
UNIQUE KEY uk_app_id (app_id)
```

### 2.4 缓存层

| 措施 | 实现方式 | 应用场景 |
|-----|---------|---------|
| **Redis原子操作** | Lua脚本/Redis事务 | 额度查询/扣减 |
| **缓存一致性** | 先写DB，再删缓存，延迟双删 | 额度同步 |
| **缓存预热** | 启动时加载热点数据 | 厂商额度 |
| **缓存失效** | 设置合理TTL | 防脏读 |

**额度缓存策略：**
```
读取：
1. 查缓存
2. 缓存命中 → 返回
3. 缓存未命中 → 查DB → 写缓存 → 返回

写入（扣减/释放）：
1. 开启DB事务
2. 更新DB（乐观锁）
3. 提交事务
4. 删除缓存
5. 延迟500ms再次删除缓存（双删）
```

**Redis Lua脚本（原子扣减）：**
```lua
local key = KEYS[1]
local amount = tonumber(ARGV[1])
local current = tonumber(redis.call('get', key) or 0)

if current >= amount then
    redis.call('decrby', key, amount)
    return 1  -- 成功
else
    return 0  -- 额度不足
end
```

### 2.5 对账层

| 措施 | 实现方式 | 应用场景 |
|-----|---------|---------|
| **实时对账** | 回调后三方状态比对 | 每单核对 |
| **定时对账** | 每5分钟扫描待支付订单查询上游 | 兜底查询 |
| **日终对账** | 每日凌晨全量订单核对 | 每日批处理 |
| **差异告警** | 状态不一致订单推送到运营群 | 及时修复 |

**对账维度：**

| 维度 | 核对内容 | 频率 |
|-----|---------|------|
| **平台-上游** | 订单状态、金额是否一致 | 实时+定时 |
| **平台-商户** | 回调是否成功、金额是否正确 | 实时 |
| **资金-订单** | 额度变动与订单是否匹配 | 日终 |
| **收益-订单** | 平台利润计算是否正确 | 日终 |

**差异订单处理：**
```
差异类型：
1. 平台有，上游无 → 人工核实，确认未支付则关单
2. 平台无，上游有 → 人工补单，补录订单数据
3. 状态不一致 → 以上游为准更新，记录原因
4. 金额不一致 → 人工核实，调账处理
```

### 2.6 监控层

| 措施 | 实现方式 | 应用场景 |
|-----|---------|---------|
| **一致性巡检** | 定时扫描异常数据 | 每小时 |
| **异常订单发现** | 规则引擎识别异常模式 | 实时 |
| **资金平衡校验** | 总额 = 可用 + 冻结 | 每日 |
| **收益核对** | 平台利润 = 商户手续费 - 上游成本 | 每日 |

**巡检规则：**

| 规则 | 说明 | 处理 |
|-----|------|------|
| 待支付订单 > 2小时 | 超时未处理订单 | 自动关单+告警 |
| 冻结额度 > 0但无待支付订单 | 额度泄漏 | 释放额度+告警 |
| 成功订单未扣减额度 | 额度扣减异常 | 人工核实+补扣 |
| 商户余额 != sum(流水) | 余额不平 | 人工核实+调账 |
| 回调失败 > 3次 | 商户接口异常 | TG告警 |

**数据一致性指标：**

| 指标 | 阈值 | 级别 |
|-----|------|------|
| 待支付订单积压 | > 100笔 | P1 |
| 差异订单数 | > 10笔/日 | P0 |
| 额度泄漏订单 | > 0笔 | P0 |
| 回调失败率 | > 5% | P1 |

## 3. 一致性保障清单汇总

| 层级 | 措施 | 核心实现 |
|-----|------|---------|
| **接口层** | 幂等Token、请求去重 | Redis SETNX、商户订单号唯一性校验 |
| **业务层** | Saga事务、状态机校验 | 订单状态机、本地事务+补偿机制 |
| **数据层** | 乐观锁/悲观锁、唯一索引 | version字段、uk_mch_order |
| **缓存层** | Redis原子操作、缓存一致性 | Lua脚本、先写DB后删缓存、延迟双删 |
| **对账层** | 多维度对账、差异告警 | 实时/定时/日终对账、差异订单推送 |
| **监控层** | 数据一致性巡检、异常订单发现 | 每小时巡检、资金平衡校验 |

---

## 4. 异常处理预案

| 异常情况 | 发现方式 | 处理方案 |
|---------|---------|---------|
| **重复支付** | 对账发现 | 人工核实，退款或调账 |
| **额度超卖** | 巡检发现 | 暂停新订单，人工核实，补扣或充值 |
| **状态卡住** | 对账发现 | 查询上游真实状态，强制更新 |
| **缓存DB不一致** | 巡检发现 | 清理缓存，从DB重新加载 |
| **重复结算** | 资金对账发现 | 冻结商户余额，追回或协商 |
| **下游重复回调** | 幂等校验发现 | 返回已处理结果，不重复结算 |

| 优先级 | 措施 | 原因 |

| P0 | 唯一索引+幂等Token | 防重复下单，资金安全第一 |
