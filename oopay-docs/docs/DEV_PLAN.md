# OOPay 支付系统开发计划

> 版本：v1.0  
> 周期：约 16 周（4个月）  
> 团队配置：后端 2-3 人，前端 1-2 人，测试 1 人

---

## 1. 开发策略

### 1.1 分阶段交付

```
MVP 阶段 (4周)     →  核心阶段 (6周)     →  完整阶段 (6周)
├── 基础框架        ├── 支付通道          ├── 对账中心
├── 商户入驻        ├── 退款功能          ├── 风控系统
├── 统一下单        ├── 账户系统          ├── 运营报表
└── 订单查询        └── 管理后台          └── 系统完善
```

### 1.2 优先级矩阵

| 优先级 | 功能 | 阶段 | 说明 |
|--------|------|------|------|
| P0 | 商户入驻/支付/查询 | MVP | 核心链路，必须可用 |
| P0 | 账户系统 | 核心 | 资金核心 |
| P1 | 退款/对账 | 核心 | 完整业务闭环 |
| P1 | 风控/通道管理 | 核心 | 运营必需 |
| P2 | 报表/权限/日志 | 完整 | 增强功能 |

## 2. 里程碑计划

### 📍 里程碑 0：准备阶段（Week 0）

**目标**：环境就绪，团队对齐

**交付物清单**：

**检查点**：
```bash
□ git clone 后能直接启动

□ 代码提交自动触发构建
```

---

### 📍 里程碑 1：MVP 交付（Week 1-4）

**目标**：核心支付链路可用

#### Week 1：基础框架 + 商户模块

**后端任务**：
```
□ 数据库初始化脚本（oopay_merchant, oopay_order 表）
□ 商户注册接口 POST /api/admin/merchant/apply
□ 商户审核接口 POST /api/admin/merchant/audit
□ API 密钥生成工具
□ 签名验证中间件
```

**前端任务**：
```
□ 登录页面
□ 商户入驻表单
□ 商户审核列表/详情
□ API 密钥查看页面
```

**每日检查点**：
```
Day 1: 数据库脚本通过，项目可启动
Day 2: 商户注册接口自测通过
Day 3: 签名验证可拦截非法请求
Day 4: 前端页面可调用接口
Day 5: 商户入驻→审核→获取密钥 完整链路测试
```

#### Week 2：支付核心（上）

**后端任务**：
```
□ 统一下单接口 POST /api/pay/create
□ 订单创建逻辑（幂等校验）
□ 订单查询接口 POST /api/pay/query
□ 订单关闭接口 POST /api/pay/close
□ oopay_order 表操作
```

**前端任务**：
```
□ 商户端 Dashboard
□ 订单列表页面
□ 订单详情页面
```

**检查点**：
```
□ 同一商户订单号24小时内幂等
□ 订单状态流转正确（PENDING→CLOSED）
□ 订单过期自动关闭
```

#### Week 3：支付核心（下）+ 通道对接

**后端任务**：
```
□ 支付通道抽象接口
□ 模拟支付通道（Mock）
□ 异步通知处理（接收回调）
□ 通知重试机制
□ 商户回调通知
```

**前端任务**：
```
□ 支付页面（二维码展示）
□ 支付结果页面
```

**检查点**：
```
□ 模拟支付通道可完成整笔交易
□ 回调通知可达商户
□ 通知失败自动重试（1/2/4/8/16分钟）
```

#### Week 4：集成测试 + MVP 交付

**测试任务**：
```
□ 商户入驻→下单→支付→回调 全流程测试
□ 并发测试（10并发下单）
□ 边界测试（超长订单号、负数金额）
□ 安全测试（签名错误、重放攻击）
```

**交付检查清单**：
```
□ 新商户可完成入驻和审核
□ 商户可成功创建支付订单
□ 支付完成后商户收到回调
□ 订单状态查询准确
□ 单元测试覆盖率 ≥ 60%
□ 接口文档与实现一致
```

**MVP 验收标准**：
```
功能：商户入驻 + 支付下单 + 回调通知
可用性：手工测试通过
性能：单接口 < 500ms
安全：签名验证 + 幂等校验
```

## 4. 风险控制

### 4.1 技术风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 第三方支付通道对接延迟 | 中 | 高 | 先实现 Mock 通道，不阻塞开发 |
| 高并发性能不达标 | 中 | 高 | Week 6 开始压测，提前优化 |
| 数据一致性问题 | 低 | 高 | 严格事务控制 + 对账机制 |

### 4.2 进度风险

| 风险 | 概率 | 影响 | 应对措施 |
|------|------|------|----------|
| 需求变更 | 高 | 中 | 冻结 MVP 需求，变更走审批 |
| 人员变动 | 低 | 高 | 关键岗位 AB 角，文档完善 |
| 第三方依赖延迟 | 中 | 中 | 预留 1 周缓冲时间 |
