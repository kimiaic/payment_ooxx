## 2. 业务定位

### 2.1 一句话描述
**USDT 结算的聚合支付路由平台** —— 作为中间层连接商户与上游支付渠道，赚取通道费率差价。

### 2.2 业务模型

```
用户付款（人民币）
    │
    ▼
商户系统 ──► 你的平台 ──► 上游支付厂商
（接口调用）   （路由/风控）   （实际通道）
                │
                ▼
        上游预付 USDT 充值额度
                │
                ▼
        你给商户结算 USDT
        （预付模式）
```

### 2.3 核心价值主张
| 利益相关方 | 价值 |
|-----------|------|
| **商户** | 一个接口对接多个支付渠道，无需分别签约微信/支付宝等 |
| **上游厂商** | 获得稳定订单流量，预付模式降低资金风险 |
| **平台** | 赚取费率差价（例如：上游成本 2%，你给商户 2.5%） |

## 3. 核心业务流程

### 3.1 资金流

**模式：预付制 + 额度管理**

1. **上游预付**：上游厂商先转账 USDT 到平台，平台为其开通对应额度
2. **额度消耗**：每笔成功订单，从上游额度中扣除订单金额 + 手续费
3. **额度告警**：上游额度低于阈值时，通知其充值
4. **商户结算**：商户预付充值 USDT，收款后余额增加，可随时提现

### 3.2 订单流

```
商户发起支付请求
    │
    ▼
平台风控检查
    │
    ▼
路由决策（选择上游）
    │
    ▼
调用上游接口
    │
    ▼
上游返回支付链接/二维码
    │
    ▼
用户完成支付
    │
    ▼
上游回调平台
    │
    ▼
平台通知商户
    │
    ▼
平台更新各方余额
```

### 3.3 结算流

| 场景 | 处理方式 |
|-----|---------|
| **上游结算** | 预付制，无需额外结算，额度用完需充值 |
| **商户结算** | 预付制，商户余额随时可提现 |
| **平台收益** | 每笔订单自动计算差价，计入平台收益账户 |

## 4. 功能需求

### 4.1 上游厂商管理

| 功能 | 说明 | 优先级 |
|-----|------|--------|
| **厂商入驻** | 上游提交资质资料，平台审核开通 | P0 |
| **额度管理** | 查看当前额度、充值记录、消耗明细 | P0 |
| **通道配置** | 配置支持的支付方式、费率、限额 | P0 |
| **权重配置** | 设置订单分配权重（负载均衡） | P1 |
| **状态管理** | 上线/下线/维护状态切换 | P0 |
| **对账功能** | 上游订单数据与平台记录比对 | P1 |

### 4.2 商户管理

| 功能 | 说明 | 优先级 |
|-----|------|--------|
| **商户入驻** | 注册、KYC 认证、审核流程 | P0 |
| **应用管理** | 一个商户可创建多个应用（App/网站/H5），独立密钥 | P0 |
| **费率配置** | 设置商户费率（必须高于上游成本） | P0 |

| **订单查询** | 查看所有订单状态、详情 | P0 |

### 4.3 订单路由系统

| 功能 | 说明 | 优先级 |
|-----|------|--------|
| **可用性筛选** | 只选择在线、额度充足的上游 | P0 |

### 4.5 对账与报表

## 6. 需求排除清单（明确不做）

从 xxpay4pro 参考后，以下功能**本期不做**：

| 功能 | 原因 |
|-----|------|
| ❌ 代理商系统 | 你直接对商户，无分销需求 |
| ❌ 核销系统 | 话费/电费/油卡核销与你业务无关 |
| ❌ ISV 服务商模式 | 你不是帮别人开发，是直接提供接口 |
| ❌ 多级分账 | 一笔订单拆给多人，你的模式是商户自己处理 |
| ❌ 法币结算 | 全部走 USDT，不碰法币清算 |
| ❌ 会员/积分系统 | 非核心业务，增加复杂度 |

## 7. 关键问题待确认

在继续细化之前，需要确认以下问题：

### 7.1 商户结算模式
- [ ] **预付制**：商户先充 USDT，收款后从预付扣，不够就停单
- [ ] **后付制**：商户先收款，每天/每周统一结算
- [ ] **混合制**：小额实时，大额后付

### 7.2 上游额度策略
- [ ] 额度用完自动停单（商户看到"通道维护"）
- [ ] 允许少量透支（平台垫资，风险高）
- [ ] 自动切换到其他上游

### 7.3 商户审核策略
- [ ] **严格**：人工审核，通过后才能使用
- [ ] **宽松**：自动通过，发现异常再封
- [ ] **分级**：小额自动，大额人工

### 7.4 汇率处理
- [ ] **实时汇率**：根据 USDT/USDT 实时汇率计算
- [ ] **固定汇率**：平台设定固定汇率，定期调整
- [ ] **浮动汇率**：平台设定基础汇率，根据市场微调

## 10. 系统模块设计

### 10.1 模块划分

基于需求分析，系统划分为 **7 个核心模块**：

```
┌─────────────────────────────────────────────────────────────┐
│                         接入层                                │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐  │
│  │ oopay-manage │  │oopay-merchant│  │    oopay-pay        │  │
│  │   运营平台   │  │   商户系统   │  │   支付网关（核心）   │  │
│  │  (Web管理端) │  │  (Web管理端) │  │  (API网关+路由)     │  │
│  └──────┬──────┘  └──────┬──────┘  └──────────┬──────────┘  │
└─────────┼────────────────┼────────────────────┼─────────────┘
          │                │                    │
          └────────────────┴────────────────────┘
                              │
          ┌───────────────────┼───────────────────┐
          │                   │                   │
          ▼                   ▼                   ▼
┌─────────────────┐  ┌──────────────┐  ┌──────────────────┐
│  oopay-service  │  │  oopay-core  │  │ oopay-z-api-base │
│  业务服务层      │  │   核心包     │  │  支付接口基础包   │
│  (核心业务逻辑)  │  │(实体/接口/工具)│  │ (SDK/签名/加密)  │
└─────────────────┘  └──────────────┘  └──────────────────┘
                              │
                              ▼
                    ┌─────────────────┐
                    │   oopay-task    │
                    │   定时任务       │
                    │ (对账/结算/清理) │
                    └─────────────────┘
```

### 10.2 模块职责定义

| 职责 | 说明 |
|-----|------|
| **实体类 (Entity)** | 订单、商户、上游、账户等所有领域模型 |
| **Dubbo 接口** | 服务间 RPC 调用接口定义 |
| **公共常量** | 状态码、枚举、配置项 |

### 10.3 模块依赖关系

```
oopay-core
    ↑
    ├── oopay-service
    │       ↑
    │       ├── oopay-manage
    │       ├── oopay-merchant
    │       ├── oopay-pay
    │       └── oopay-task
    │
    └── oopay-z-api-base （独立，供外部使用）
```

**依赖原则：**
- 上层模块可调用下层模块
- 禁止跨层调用（如 manage 直接调用 core，不经过 service）
- 禁止循环依赖
- oopay-z-api-base 是独立 SDK，不依赖平台内部模块
