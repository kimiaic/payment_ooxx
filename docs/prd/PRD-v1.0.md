> 状态：定稿 | 更新：2026-04-07

---

## 1. 业务定位

**USDT 结算的聚合支付路由平台** — 连接商户与上游支付渠道，赚取费率差价。

### 1.1 核心流程

```
用户付款(CNY) → 商户 → 平台(路由/风控) → 上游厂商 → 上游回调 → 平台结算USDT给商户
```

### 1.2 资金模式

| 角色 | 模式 | 说明 |
|-----|------|------|
| 上游厂商 | 预付制 | 先充 USDT 获得额度，订单成功扣额度 |
| 商户 | 混合制 | 可预付(负余额)，可实时提现，每日结算总结 |
| 平台 | 差价 | 商户费率(2.5%) - 上游费率(2.0%) = 利润 |

## 3. 核心概念模型

### 3.1 四层结构

```
┌─────────────────────────────────────────┐
│  支付产品（商户可见）                     │
│  oopayalih5 → OOpay支付宝H5              │
│  费率：2.5%                              │
└─────────────────┬───────────────────────┘
                  │ 路由规则（权重/筛选）
┌─────────────────▼───────────────────────┐
│  上游通道（平台内部）                     │
│  好运支付-支付宝H5（费率2.0%，权重60）     │
│  好来支付-支付宝H5（费率2.2%，权重40）     │
└─────────────────┬───────────────────────┘
                  │ 共享额度
┌─────────────────▼───────────────────────┐
│  厂商（额度主体）                         │
│  好运支付：总额度10000 USDT               │
└─────────────────────────────────────────┘
```

### 3.2 概念定义

| 概念 | 说明 | 示例 |
|-----|------|------|
| 厂商 | 上游公司主体，额度归属 | 好运支付、好来支付 |
| 上游通道 | 厂商下的支付方式 | 好运支付-支付宝H5 |
| 支付产品 | 商户对接的统一入口 | oopayalih5 |
| 路由规则 | 订单分配策略 | 权重60:40，失败自动切换 |

## 4. 关键业务规则

### 4.1 额度管理

| 环节 | 规则 |
|-----|------|
| 充值 | 厂商充 USDT → 增加额度 |
| 冻结 | 下单时冻结对应额度 |
| 扣减 | 支付成功正式扣减 |
| 释放 | 30分钟超时未支付自动释放 |
| 预警 | 可用额度 < 1000 CNY 时TG告警 |

### 4.2 路由规则

- 筛选：在线 + 额度充足 + 限额匹配 + IP白名单通过
- 分配：按权重比例随机分配
- 切换：当前通道失败/额度不足时自动切换备用通道
- 熔断：全部通道不可用时返回"通道维护"

### 4.3 商户准入

- 邀请制：邀请码注册，无需审核，邀请人背书
- IP白名单：下单服务器IP必须在白名单内
- 产品授权：运营为每个商户单独配置可用产品及费率

### 4.4 结算规则

- 商户可负余额（预付），可正余额（可提现）
- 每日结束推送余额总结
- 运营可在后台手动调整商户/厂商余额

### 4.5 汇率规则

- 来源：OKX C2C 实时汇率第一档（USDT/CNY）
- 更新：每5分钟抓取
- 汇差：默认上浮0.1，上游和商户可分别设置
- 示例：OKX 7.20 + 汇差 0.1 = 结算汇率 7.30

### 4.6 回调与兜底

- 回调为主：上游主动通知支付结果
- 兜底查询：每5分钟查询待支付订单，2小时后停止
- 重试：回调失败时1min/5min/15min重试3次

## 5. 支付接口

### 5.1 统一下单

```
POST /api/pay/createOrder

请求：
{
  "mchId": "M001",           // 商户号
  "appId": "APP001",         // 应用ID
  "productId": "oopayalih5", // 支付产品
  "mchOrderNo": "ORDER_001", // 商户订单号
  "currency": "CNY",
  "amount": 100000,          // 金额(分)
  "clientIp": "192.168.1.1",
  "subject": "商品标题",
  "notifyUrl": "https://商户.com/notify",
  "returnUrl": "https://商户.com/return",
  "sign": "MD5签名"
}

响应：
{
  "retCode": "SUCCESS",
  "data": {
    "payOrderId": "OO20240407123456",
    "status": "0",             // 0-生成,1-支付中,2-成功,3-完成,4-关闭
    "payData": "{\"payUrl\":\"https://...\"}"
  }
}
```

### 5.2 回调通知

```
POST 商户notifyUrl

{
  "payOrderId": "OO20240407123456",
  "mchOrderNo": "ORDER_001",
  "status": "2",               // 2-支付成功
  "amount": 100000,
  "paySuccTime": 1712476800,
  "sign": "MD5签名"
}

商户响应：SUCCESS（大写）
```

### 5.3 签名算法

1. 参数按ASCII升序排序
2. 拼接：`key1=value1&key2=value2&key=商户密钥`
3. MD5加密，转大写

### 6.1 运营后台（oopay-manage）

| 模块 | 功能 |
|-----|------|
| 厂商管理 | 入驻、额度调整、状态管理 |
| 通道管理 | 费率、限额、权重配置 |

**异常订单操作：**
- 手动补单：确认已付款但未回调
- 强制关单：异常订单关闭，释放额度
- 调账：手动调整余额
- 查询上游：主动核实订单状态

## 7. TG Bot 通知系统

### 7.1 部署方式

独立部署，通过HTTP/RPC与主系统通信。

### 7.2 通知场景

| 场景 | 触发条件 | 接收群 |
|-----|---------|--------|
| 额度预警 | 厂商可用 < 1000 CNY | 厂商合作群 |
| 每日账单 | 每日00:30 | 商户合作群 |
| 回调失败 | 回调3次失败 | 商户合作群 |
| 额度增加 | 手动调整后 | 运营指定群 |
| 异常订单 | 对账差异 | 运营内部群 |
| 通道熔断 | 连续10次失败 | 运营内部群 |

### 7.3 Bot指令

| 指令 | 功能 | 权限 |
|-----|------|------|
| `/addbalance 商户 金额 [备注]` | 给商户增加预付款 | 财务/管理员 |
| `/addprovider 厂商 金额 [备注]` | 给厂商增加额度 | 财务/管理员 |
| `/check 商户` | 查询商户余额和昨日数据 | 运营/财务 |
| `/checkprovider 厂商` | 查询厂商额度 | 运营/财务 |
| `/help` | 帮助信息 | 所有人 |

**安全机制：**
- 金额 > 10000 USDT 需二次确认（6位验证码私聊发送）
- 指令只能在指定群组使用
- 操作人需在白名单

## 9. 容灾设计

### 9.1 上游故障

- 检测：连续5笔失败标记疑似故障，10笔熔断
- 切换：自动切换到备用通道
- 恢复：连续3笔成功自动恢复，或手动恢复
- 兜底：可用上游<2个时紧急告警

### 9.2 商户侧

- 通道不可用时返回 `CHANNEL_UNAVAILABLE`
- 商户自行路由切换或暂停支付
