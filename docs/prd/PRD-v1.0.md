## 2. 业务定位

### 2.1 一句话描述
**USDT 结算的聚合支付路由平台** —— 作为中间层连接商户与上游支付渠道，赚取通道费率差价。

### 2.2 业务模型

```
用户付款（人民币）
    │
    ▼
商户系统 ──► 你的平台 ──► 上游支付厂商
（接口调用）   （路由/风控）   （实际通道）
                │
                ▼
        上游预付 USDT 充值额度
                │
                ▼
        你给商户结算 USDT
        （预付模式）
```

### 2.3 核心价值主张
| 利益相关方 | 价值 |
|-----------|------|
| **商户** | 一个接口对接多个支付渠道，无需分别签约微信/支付宝等 |
| **上游厂商** | 获得稳定订单流量，预付模式降低资金风险 |
| **平台** | 赚取费率差价（例如：上游成本 2%，你给商户 2.5%） |

## 3. 核心业务流程

### 3.1 资金流

**模式：预付制 + 额度管理**

1. **上游预付**：上游厂商先转账 USDT 到平台，平台为其开通对应额度
2. **额度消耗**：每笔成功订单，从上游额度中扣除订单金额 + 手续费
3. **额度告警**：上游额度低于阈值时，通知其充值
4. **商户结算**：商户预付充值 USDT，收款后余额增加，可随时提现

### 3.2 订单流

```
商户发起支付请求
    │
    ▼
平台风控检查
    │
    ▼
路由决策（选择上游）
    │
    ▼
调用上游接口
    │
    ▼
上游返回支付链接/二维码
    │
    ▼
用户完成支付
    │
    ▼
上游回调平台
    │
    ▼
平台通知商户
    │
    ▼
平台更新各方余额
```

### 3.3 结算流

| 场景 | 处理方式 |
|-----|---------|
| **上游结算** | 预付制，无需额外结算，额度用完需充值 |
| **商户结算** | 预付制，商户余额随时可提现 |
| **平台收益** | 每笔订单自动计算差价，计入平台收益账户 |

## 6. 需求排除清单（明确不做）

从 xxpay4pro 参考后，以下功能**本期不做**：

| 功能 | 原因 |
|-----|------|
| ❌ 代理商系统 | 你直接对商户，无分销需求 |
| ❌ 核销系统 | 话费/电费/油卡核销与你业务无关 |
| ❌ ISV 服务商模式 | 你不是帮别人开发，是直接提供接口 |
| ❌ 多级分账 | 一笔订单拆给多人，你的模式是商户自己处理 |
| ❌ 法币结算 | 全部走 USDT，不碰法币清算 |
| ❌ 会员/积分系统 | 非核心业务，增加复杂度 |

## 7. 关键问题待确认

在继续细化之前，需要确认以下问题：

### 7.1 商户结算模式
- [ ] **预付制**：商户先充 USDT，收款后从预付扣，不够就停单
- [ ] **后付制**：商户先收款，每天/每周统一结算
- [ ] **混合制**：小额实时，大额后付

### 7.2 上游额度策略
- [ ] 额度用完自动停单（商户看到"通道维护"）
- [ ] 允许少量透支（平台垫资，风险高）
- [ ] 自动切换到其他上游

### 7.3 商户审核策略
- [ ] **严格**：人工审核，通过后才能使用
- [ ] **宽松**：自动通过，发现异常再封
- [ ] **分级**：小额自动，大额人工

### 7.4 汇率处理
- [ ] **实时汇率**：根据 USDT/USDT 实时汇率计算
- [ ] **固定汇率**：平台设定固定汇率，定期调整
- [ ] **浮动汇率**：平台设定基础汇率，根据市场微调

## 10. 系统模块设计

### 10.1 模块划分

基于需求分析，系统划分为 **7 个核心模块**：

```
┌─────────────────────────────────────────────────────────────┐
│                         接入层                                │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────────┐  │
│  │ oopay-manage │  │oopay-merchant│  │    oopay-pay        │  │
│  │   运营平台   │  │   商户系统   │  │   支付网关（核心）   │  │
│  │  (Web管理端) │  │  (Web管理端) │  │  (API网关+路由)     │  │
│  └──────┬──────┘  └──────┬──────┘  └──────────┬──────────┘  │
└─────────┼────────────────┼────────────────────┼─────────────┘
          │                │                    │
          └────────────────┴────────────────────┘
                              │
          ┌───────────────────┼───────────────────┐
          │                   │                   │
          ▼                   ▼                   ▼
┌─────────────────┐  ┌──────────────┐  ┌──────────────────┐
│  oopay-service  │  │  oopay-core  │  │ oopay-z-api-base │
│  业务服务层      │  │   核心包     │  │  支付接口基础包   │
│  (核心业务逻辑)  │  │(实体/接口/工具)│  │ (SDK/签名/加密)  │

```

| 职责 | 说明 |
|-----|------|
| **Dashboard** | 核心数据看板 |

| **商户管理** | 邀请审核、产品授权、独立费率设置、冻结/解冻 |
| **订单管理** | 全平台订单查询、异常处理、手动补单 |
| **资金管理** | 厂商/商户余额调整、结算审批、流水查询 |
| **风控管理** | IP 白名单、黑名单管理、风险订单处理 |

| **操作日志** | 管理员操作审计 |

**依赖：** oopay-core, oopay-service  
**用户：** 平台运营人员、管理员

**关键功能详解：**

**1. 产品管理**
- 运营人员可**无限增设**支付产品
- 每个产品需配置：名称、编码、支付方式类型、默认费率
- 创建产品时**强制提示**：必须配置底层上游通道和路由规则
- 产品状态：草稿 / 上线 / 下线

**2. 商户管理**
- **邀请制入驻：** 生成邀请码 → 商户注册 → 人工审核
- **产品授权：** 为每个商户单独配置可使用的支付产品列表
- **独立费率：** 商户可继承产品默认费率，也可单独设置
- **IP 白名单：** 配置商户允许下单的服务器 IP 列表

**示例配置：**
```
商户: 张三科技
├─ 可用产品:
│   ├─ oopayalih5 (支付宝H5) - 费率 2.5% (继承默认)
│   ├─ oopaywxh5 (微信H5) - 费率 2.3% (自定义优惠)
│   └─ oopayalisc (支付宝扫码) - 费率 2.5% (继承默认)
│
├─ IP 白名单:
│   ├─ 192.168.1.100
│   └─ 10.0.0.50
│
└─ 结算地址: Txxxxxxxxx (USDT-TRC20)
```

#### oopay-task（定时任务 + 消息推送）

**核心职责：**

| 类别 | 职责 | 说明 |
|-----|------|------|
| **对账任务** | 定时与上游对账，发现差异订单 | 每5分钟查询待支付订单状态，2小时后停止 |
| **结算任务** | 自动结算商户余额到 USDT 地址 | 触发自动结算、生成结算记录 |
| **数据清理** | 清理过期日志、备份历史数据 | 订单数据保留5年，日志保留90天 |
| **统计汇总** | 每日/周/月报表生成 | 商户账单、平台收益报表 |
| **消息推送** | TG Bot 通知（见下文）| 额度预警、账单推送、回调失败提醒 |

**依赖：** oopay-core, oopay-service  
**部署：** 单节点部署（避免重复执行）

### 11.3 关键业务规则

#### 规则 1：额度管理
- **充值维度：** 厂商维度（好运支付充 10000 USDT，所有旗下通道共享）
- **消耗维度：** 订单维度（每成功一笔，从厂商额度扣除订单金额）
- **冻结机制：**
  - 用户下单后**冻结额度**
  - **支付成功后正式扣减**额度
  - **超时未支付自动释放**额度（默认 30 分钟，可配置）

#### 规则 2：路由选择
- **筛选条件：** 在线状态 + 额度充足 + 限额匹配 + 风控通过
- **权重分配：** 通过的产品按比例分配订单
- **失败重试：** 主通道失败后，自动尝试次通道（不占用权重）

#### 规则 3：产品映射
- **一对多：** 一个产品可以映射多个上游通道
- **支付方式匹配：** 产品（支付宝H5）只能映射同类型通道（支付宝H5）
- **费率差异：** 产品费率（2.5%）> 上游费率（2.0%），差价是平台利润

#### 规则 4：商户准入与风控
- **邀请制：** 商户注册需邀请码，人工审核后开通
- **IP 白名单：** 商户下单服务器 IP 必须在白名单内
- **风控检查点：**
  - 商户入驻：人工审核（邀请制）
  - 商户下单：IP 白名单校验
  - 其他环节：根据后续需求扩展

#### 规则 5：结算规则
- **厂商结算（预付制）：**
  - 厂商先充值 USDT 到平台，获得对应额度
  - 每笔订单成功，从厂商额度扣除订单金额
  - 后台支持运营人员**手动调整**厂商额度（补录线下充值）

- **商户结算（实时+预付）：**
  - **预付模式：** 商户可先欠费（余额为负），当欠款达到阈值时提醒结算
  - **实时结算：** 商户余额为正时，可随时申请提现
  - **手动结算：** 运营人员可在后台手动为商户增加结算金额

#### 规则 6：订单查询与状态
- **商户可见状态（简版）：**
  - `待支付` - 订单创建，等待用户付款
  - `已支付` - 支付成功，待结算
  - `已关闭` - 超时未支付或主动关闭

#### 规则 7：上游回调与兜底查询
- **回调为主：** 上游支付成功后主动回调平台
- **定时查询兜底：**
  - 频率：每 5 分钟查询一次待支付订单
  - 超时：订单创建后 2 小时不再查询（已超时关闭）
  - 联动：查询到支付成功，立即扣减额度并更新订单状态

## 12. 通知与消息系统（TG Bot）

### 12.1 系统定位

TG Bot 作为**运营协同工具**，实现：
- **实时通知：** 关键业务事件即时推送
- **移动办公：** 运营人员无需登录后台即可处理事务
- **群组协作：** 相关方在同一群组接收信息，便于沟通

### 12.2 群组体系设计

| 群组类型 | 成员 | 用途 |
|---------|------|------|
| **厂商合作群** | 厂商联系人 + 平台运营 | 额度预警、充值确认、技术对接 |
| **商户合作群** | 商户联系人 + 平台运营 | 每日账单、回调失败提醒、结算通知 |
| **运营内部群** | 平台运营 + 财务 + 技术 | 系统告警、异常订单、统计数据 |
| **管理指令群** | 管理员 + 财务（核心人员）| Bot 指令操作、敏感操作确认 |

**配置方式：**
- 在 `oopay-manage` 后台配置每个厂商/商户绑定的 Telegram Group ID
- Bot 需要被添加到对应群组并授予发送消息权限

## 14. 技术架构选型

### 14.1 数据库选型：PostgreSQL

**选择理由：**
- **ACID 严格性**：相比 MySQL，PostgreSQL 的事务隔离级别更严格，适合金融场景
- **数据完整性**：支持更复杂的约束和触发器，减少数据不一致风险
- **JSON 支持**：原生 JSON/JSONB 类型，便于存储灵活的配置数据
- **扩展性**：支持分区表、并行查询，应对数据增长

**数据库设计原则：**
- 订单表按时间分区（按月/按日）
- 关键字段加索引（订单号、商户号、状态、时间）
- 敏感数据加密存储（USDT 地址、密钥）

### 14.2 缓存策略

**厂商额度处理：**
- **读策略**：优先查缓存，缓存未命中查 DB
- **写策略**：
  - 额度扣减：先写 DB，成功后更新缓存（保证数据不丢）
  - 额度增加：先写 DB，成功后更新缓存
- **一致性**：缓存与 DB 最终一致（秒级延迟可接受）
- **缓存过期**：5 分钟，防止长期不一致

**其他缓存场景：**
- 路由配置：缓存 1 小时，变更时主动刷新
- 商户信息：缓存 10 分钟
- 费率配置：缓存 1 小时

### 14.3 服务通信方式

**采用 Dubbo RPC（参考 xxpay4pro）：**
- 高性能：二进制序列化，比 HTTP 更快
- 服务治理：内置负载均衡、容错、服务降级
- 接口定义：通过接口契约明确服务边界
- 注册中心：ZooKeeper / Nacos

**模块间通信：**
- `oopay-pay` → `oopay-service`：同步 RPC 调用
- `oopay-manage` → `oopay-service`：同步 RPC 调用
- 消息推送：异步 Redis List（简单可靠）

### 14.4 消息队列选型：Redis List

**选择理由：**
- **简单**：无需额外部署 RabbitMQ/RocketMQ
- **够用**：当前场景主要是异步通知，无复杂路由需求
- **可靠**：配合持久化，数据不丢
- **成本**：减少运维复杂度

**使用场景：**
- TG Bot 消息推送队列
- 回调商户重试队列
- 对账任务队列

**数据结构：**
```
queue:bot:message    ← Bot 消息队列
queue:callback       ← 回调重试队列
queue:reconcile      ← 对账任务队列
```

| amount | Long | 金额（分）|
| status | String | 订单状态：0-订单生成, 1-支付中, 2-支付成功, 3-业务处理完成, 4-已关闭 |

### 15.2 回调通知格式

**回调方式：** HTTP POST，Content-Type: application/json

**回调参数：**

| 字段 | 类型 | 说明 |
|-----|------|------|
| payOrderId | String | 平台订单号 |
| mchId | String | 商户号 |
| appId | String | 应用 ID |
| productId | String | 支付产品编码 |
| mchOrderNo | String | 商户订单号 |
| currency | String | 货币类型，CNY |
| amount | Long | 金额（分）|
| status | String | 订单状态：2-支付成功 |
| paySuccTime | Long | 支付成功时间戳 |
| backType | String | 通知类型：1-前端跳转, 2-后台通知 |
| sign | String | 签名 |

**商户响应要求：**
- 成功：返回字符串 `SUCCESS`（大写，无引号）
- 失败：返回其他内容，平台会重试

**重试机制：**
- 间隔：1 分钟、5 分钟、15 分钟
- 最多重试 3 次
- 3 次均失败，发送到商户合作群告警

### 15.3 订单查询接口

**接口地址：** `POST /api/pay/queryOrder`

**请求参数：**

| 字段 | 类型 | 必填 | 说明 |
|-----|------|------|------|
| mchId | String | 是 | 商户号 |
| payOrderId | String | 否 | 平台订单号（与 mchOrderNo 二选一）|
| mchOrderNo | String | 否 | 商户订单号 |
| sign | String | 是 | 签名 |

**返回参数：** 同创建订单接口

### 15.4 签名算法

**签名步骤：**
1. 将所有非空参数按字段名 ASCII 升序排序
2. 拼接成 `key1=value1&key2=value2` 格式
3. 最后拼接 `&key=商户密钥`
4. 对拼接字符串进行 MD5 加密，转大写

**示例：**
```
原始参数：mchId=M001, amount=1000, productId=oopayalih5
排序后：amount=1000&mchId=M001&productId=oopayalih5
拼接密钥：amount=1000&mchId=M001&productId=oopayalih5&key=商户密钥
MD5 加密：E10ADC3949BA59ABBE56E057F20F883E
```

━━━━━━━━━━━━━━━

**预警阈值设置：**

- 按厂商单独配置，默认 1000 CNY

### 16.2 异常订单处理

运营后台「订单管理」页面，异常订单可操作：

| 操作 | 功能 | 适用场景 |
|-----|------|---------|
| **手动补单** | 标记订单为已支付，扣减额度，通知商户 | 上游回调丢失，但确认用户已付款 |
| **强制关单** | 标记订单为已关闭，释放冻结额度 | 订单异常，确认无法完成 |
| **调账** | 手动调整商户/厂商余额 | 差错处理、补偿 |
| **查询上游** | 主动查询上游订单状态 | 状态不一致时核实 |

**操作流程：**
```
订单详情页
    │
    ▼
查看三方状态对比（平台 vs 上游 vs 商户）
    │
    ▼
选择操作：补单 / 关单 / 调账 / 查询上游
    │
    ▼
填写原因，二次确认
    │
    ▼
执行操作，记录日志，通知相关方
```

### 16.3 权限管理设计

**角色体系（V1 版本）：**

| 角色 | 权限范围 |
|-----|---------|
| **超级管理员** | 全部权限，可管理其他账号 |
| **财务** | 资金管理、结算审批、额度调整、查看报表 |
| **运营** | 商户管理、订单管理、查看数据（只能看到分配给自己的商户）|
| **客服** | 查看订单、处理客诉（只读）|

**数据权限控制：**
- 商户数据：运营只能看到「分配给自己」的商户
- 订单数据：按商户归属查看
- 资金数据：仅财务和管理员可操作

**后期扩展：**
- V2 版本增加「权限管理页面」
- 支持自定义角色
- 支持按功能点细粒度授权

## 18. 容灾与监控

### 18.1 上游故障处理

**故障检测：**
- 连续失败 5 笔，标记为「疑似故障」
- 连续失败 10 笔，自动熔断，状态改为「维护」
- TG Bot 推送告警到运营群

**故障恢复：**
- 自动探测：每 5 分钟检测一次，连续成功 3 笔自动恢复
- 手动恢复：运营在后台点击「恢复通道」

**商户侧处理：**
```
商户调用平台接口
        │
        ▼
平台返回错误：CHANNEL_UNAVAILABLE（通道不可用）
        │
        ▼
商户内部路由机制：自动切换到其他支付通道
        │
        ▼
如果平台全部通道不可用，商户暂停支付功能
```

**平台兜底机制：**
- 监控所有上游健康状态
- 当可用上游 < 2 个时，发送紧急告警
- 运营紧急联系备用上游启用

### 18.3 TG Bot 部署方式

- 部署：Docker 容器，独立扩缩容

### 18.4 确认码机制

**大额操作二次确认：**
- 触发条件：操作金额 > 10000 USDT
- 确认码：随机 6 位数字（123456）
- 发送方式：Bot 私聊发送给操作人
- 确认方式：在群组回复 `/confirm 123456`

**流程：**
```
运营发送：/addprovider haoyun 15000 大额充值
Bot 回复：⚠️ 金额超过 10000 USDT，需要二次确认
         确认码已私聊发送，请查收
         在群组回复 /confirm 确认码 完成操作

运营私聊收到：【OOpay】您的确认码是：847291

运营在群组发送：/confirm 847291
Bot 回复：✅ 操作成功，厂商好运支付额度增加 15000 USDT
```
