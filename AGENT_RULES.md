## 一、记忆与上下文管理

### 1.1 每次开始工作前必做

```
1. 读取本文件（AGENT_RULES.md）
2. 读取 CHANGELOG.md 了解最新版本状态
3. 读取 TODO.md 了解当前待处理事项
4. 汇报当前理解，等待确认后再执行
```

### 1.2 记忆衰减防止机制

- 所有决策、规则、已完成事项**必须写入文件**，不依赖对话记忆
- 每次对话开始视为全新上下文，必须从文件中恢复状态
- 重要决定（架构调整、安全要求变更）必须同步更新到对应文档
- 不得说"上次我们说好的"——上次说好的必须已经写进文件

### 1.3 状态文件职责

| 文件 | 职责 | 更新时机 |
|-----|------|---------|
| `AGENT_RULES.md` | 协作规则（本文件）| 规则变更时 |
| `CHANGELOG.md` | 版本变更记录 | 每次提交后 |
| `TODO.md` | 当前待处理事项 | 每次工作结束时 |
| `FULL_DEV_PLAN.md` | 开发阶段计划与进度 | Phase 状态变更时 |

## 二、Git 提交规范

### 2.1 提交格式

```
类型(范围): 简短说明

[可选正文：说明为什么这样改]
```

### 2.2 类型定义

| 类型 | 用途 | 示例 |
|-----|------|------|
| `docs` | 文档新增或内容修改 | `docs(db): 补充 notify_log 表字段` |
| `fix` | 修复错误内容或链接 | `fix(issue): 修正 #1 中计划文档链接路径` |
| `refactor` | 结构调整，内容不变 | `refactor: 删除废弃 docs/ 目录` |
| `feat` | 新增功能文档或规划 | `feat(api): 新增管理后台接口文档` |
| `security` | 安全相关修改 | `security: 补充 timestamp 防重放要求` |
| `chore` | 配置、工具类文件 | `chore: 更新 .gitignore` |

### 2.3 提交禁止事项

- ❌ 禁止提交空文件或只有占位符的文件（如只有标题没有内容）
- ❌ 禁止一次提交混入多个不相关的改动
- ❌ 禁止提交 `test.md`、`temp.md`、`draft.md` 等临时文件
- ❌ 禁止提交自动生成的无意义文件（如 `.DS_Store`、`node_modules`）
- ❌ 禁止在没有实质内容变更的情况下提交（如只改了空格）
- ❌ 禁止提交信息使用"update"、"fix"、"修改"等模糊描述

### 2.4 每次提交前自检

```
□ 这次提交解决了什么具体问题？
□ 是否产生了新的多余文件？
□ 提交信息是否清晰描述了改动内容？
□ 是否有临时文件需要清理？
□ CHANGELOG.md 是否已更新？
```

## 三、任务执行规范

### 3.1 禁止幻想，禁止虚报

- **禁止**在未实际执行的情况下汇报"已完成"
- **禁止**编造文件内容、Commit SHA、测试结果
- **禁止**假设某个文件存在而不实际读取验证
- 如果无法完成某项任务，必须明确说明原因，不得绕过

### 3.2 任务开始前

```
1. 明确理解任务目标
2. 列出将要改动的文件清单
3. 确认没有歧义后再执行
4. 遇到不确定的地方，先问，不要猜
```

### 3.3 任务执行中

```
1. 每完成一个子任务，实际验证结果
2. 发现预期外的问题，立即汇报，不自行决策
3. 不得跨越授权范围执行未被指定的任务
4. 修改核心文档（PRD、安全设计）前必须得到明确指令
```

### 3.4 任务完成标准

满足以下全部条件才算"有效完成"：

```
□ 文件实际存在于仓库，可访问
□ 内容与任务要求一致，无占位符
□ 相关联文件已同步更新（如链接、目录索引）
□ CHANGELOG.md 已记录本次变更
□ TODO.md 已移除该事项
□ 已提交到 GitHub 并可验证
```

## 四、文档规范

### 4.1 目录结构（权威版本）

```
payment_ooxx/
├── AGENT_RULES.md ← 本文件，AI 协作规则
├── CHANGELOG.md ← 版本变更记录
├── CONTRIBUTING.md ← 人工开发规范
├── FULL_DEV_PLAN.md ← 完整开发计划
├── TODO.md ← 当前待处理事项
├── README.md ← 项目总览
└── oopay-docs/ ← 唯一文档目录
 └── docs/
 ├── prd/ ← 产品需求
 ├── architecture/ ← 系统架构
 ├── db/ ← 数据库设计
 ├── security/ ← 安全设计
 ├── api/ ← 接口文档
 ├── ops/ ← 运营手册
 └── ui/ ← UI 设计
```

**注意：**
- 根目录下的 `docs/` 目录已废弃，禁止再往里写内容
- 所有新文档必须放入 `oopay-docs/docs/` 对应子目录
- 不得创建上述结构之外的新目录，如有必要先汇报

### 4.2 文件命名规范

- 全部小写，单词用 `-` 连接
- 有版本的文件加版本号：`PRD-v1.0.md`
- 禁止使用：`test`、`temp`、`draft`、`new`、`final`、`copy` 等词

### 4.3 文档内容规范

- 每份文档顶部必须有：文件用途说明、版本、更新日期
- 同一内容只能出现在一个文件（单一事实来源）
- 发现重复内容：保留权威版本，其他改为引用链接
- 禁止在文档中留下空章节、TODO 占位符、"待补充"字样

## 五、安全红线

以下要求是硬性规定，**任何阶段都不得违反**：

| 要求 | 说明 |
|-----|------|
| 签名必须含 timestamp | 服务端拒绝超过 5 分钟的请求，防重放攻击 |
| 密码必须 bcrypt | 禁止 MD5/SHA1 存储任何密码，cost factor ≥ 12 |
| Secret Key 加密存储 | AES-256-GCM，密钥通过环境变量注入 |
| TG Bot 大额阈值 | 1,000 USDT（非原 PRD 中的 10,000）|
| 敏感信息不入库 | 密钥、密码、Token 禁止出现在任何文档或代码中 |

| 任务描述 | 具体文件路径 | ✅ 已完成 |

## 验证结果
- 文件已存在：[链接或路径]
- 内容符合要求：是/否（如否说明原因）
- CHANGELOG 已更新：是/否

## 当前遗留问题
- 问题1：描述
- 问题2：描述
（如无遗留问题，写"无"）

## 下一步建议

## 八、违规处理

发现以下行为，立即停止当前任务并汇报：

- 虚报完成状态

- 发现文档内容与实际不符

**原则：做错了说出来，比假装做对了要好。**
